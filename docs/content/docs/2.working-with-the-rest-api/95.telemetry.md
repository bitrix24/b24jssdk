The SDK appends three telemetry query parameters to every outgoing REST request. Their names are stable across versions; their values let you correlate SDK requests on the Bitrix24 side and on your own infrastructure (logs, traces, dashboards).

| Parameter | Source | Notes |
|----|----|----|
| `bx24_request_id` | The `requestId` you pass to `actions.v{2,3}.*.make({ requestId })`, or a UUID v7 generated by [`RequestIdGenerator`](/docs/working-with-the-rest-api/core-request-id-generator/) when omitted. | Sent as a query parameter and as an `X-Request-ID` header. All three telemetry params are **skipped for the legacy positional `task.*` methods** (`task.commentitem.*`, `task.checklistitem.*`, …) on **both** REST API v2 and v3 — those methods read the query string positionally, so Bitrix24 rejects the call (`WRONG_ARGUMENTS`) when the params are present. Modern `tasks.task.*` is not affected by the constraint. |
| `bx24_sdk_ver` | Build-time constant, replaced by `unbuild`. Equal to the package's `package.json` version. | Lets Bitrix24 distinguish issues by SDK version. |
| `bx24_sdk_type` | Build-time constant. Currently `b24-js-sdk`. | Distinguishes `b24jssdk` from other SDKs (PHP, Python). |

## Header Name

The same `bx24_request_id` value is also placed in the `X-Request-ID` header (`DEFAULT_REQUEST_ID_HEADER_FIELD_NAME` in [`RequestIdGenerator`](/docs/working-with-the-rest-api/core-request-id-generator/)). If your edge layer logs `X-Request-ID`, you'll see the same id in your logs and in the Bitrix24 server logs.

## Customising the Generator

If you need to inherit an upstream tracing id (e.g. propagate `X-Request-ID` from your gateway), implement [`IRequestIdGenerator`](/docs/working-with-the-rest-api/core-request-id-generator/) and inject it through a custom HTTP setup. Each call must return a unique value — the limiter and the request log key on it.

## Always Pass `requestId`

Even though the SDK generates one for you, **always set `requestId` explicitly** in production code:

```ts
await $b24.actions.v2.call.make({
  method: 'crm.contact.get',
  params: { id: 123 },
  requestId: `contact-${123}-${Date.now()}`
})
```

Reasons:

- The id appears in the SDK's logger output, simplifying correlation.
- The id is used by `RestrictionManager` for retry / backoff bookkeeping.
- A meaningful id is far easier to grep in Bitrix24 support tickets than a UUID.
