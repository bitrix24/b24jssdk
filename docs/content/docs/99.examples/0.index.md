This section ships a small catalogue of self-contained recipes — each one a single TypeScript file you can run with `tsx` after setting `B24_HOOK` to your incoming webhook URL. They use the canonical `b24.actions.v{2,3}.*.make()` surface (the legacy `b24.callMethod` / `callBatch` / `callListMethod` / `fetchListMethod` is `@deprecated` for `2.0.0`).

The same recipes also live under `.claude/skills/b24jssdk-recipes/examples/` so that AI agents can read them as authoritative templates.

## Common boot snippet

Every recipe starts with the same loader:

```ts
import { B24Hook, LoggerBrowser, type TypeB24 } from '@bitrix24/b24jssdk'

export function bootB24(): TypeB24 {
  const url = process.env.B24_HOOK
  if (!url) throw new Error('B24_HOOK env var is required')
  const $b24 = B24Hook.fromWebhookUrl(url)
  $b24.offClientSideWarning?.()
  return $b24
}

export const logger = LoggerBrowser.build('Recipe', process.env.NODE_ENV !== 'production')
```

Functions inside each recipe accept `$b24: TypeB24`, so the same code runs unchanged with `B24Frame` (in-iframe app) or `B24OAuth` (OAuth-installed app). Only the boot step changes.

## API-version split

| Method | Action surface |
|---|---|
| `crm.item.{get,list,add,update,delete}` | `actions.v2.*` |
| `crm.activity.list`, `crm.timeline.comment.add`, `disk.*`, `im.*`, `user.*`, `profile` | `actions.v2.*` |
| `tasks.task.{add,get,update,delete}` | `actions.v3.*` |
| `tasks.task.list` | `actions.v2.*` |
| `main.eventlog.{list,get,tail}` | `actions.v3.*` |

> Calling a v3-eligible method via `actions.v2.*` works but logs a warning (`JSSDK_CORE_METHOD_AVAILABLE_IN_API_V3`). Calling a non-v3 method via `actions.v3.*` throws `SdkError(JSSDK_CORE_METHOD_NOT_SUPPORT_IN_API_V3)`.

## Notes that apply to every recipe

- CRM `crm.item.*` methods use camelCase fields (`stageId`, `assignedById`, `opportunity`). Tasks (`tasks.task.*`), Disk (`disk.*`), and IM (`im.*`) use classic uppercase fields. Recipes follow this split.
- Stage IDs on multi-funnel portals carry a category prefix (`C2:WON`, `C4:LOSE`). Recipes 1, 3, and 6 strip it via a small `baseStage()` helper.
- `callList.make` / `fetchList.make` **strip user-supplied `order`** — keyset pagination requires `idKey ASC`. Narrow with `filter` instead.
- For `crm.item.list` set `idKey: 'id'` and `customKeyForResult: 'items'`. For classic methods use the default `idKey: 'ID'` (uppercase) and either omit `customKeyForResult` or set it to the right key (`'tasks'`, `'files'`, …).
- Filters use the **v2 prefix** dialect (`'>=opportunity'`, `'!stageId'`, `'%title'`) because every recipe stays on v2 actions for CRM. MongoDB-style `$gt`/`$ne`/`$contains` is a VibeCode-only convention and is **not** valid against Bitrix24 REST.
- Bitrix24 outbound webhooks always need a `200` response — recipe 7 always returns 200, even when the handler throws, so Bitrix24 doesn't retry for 24 h.
