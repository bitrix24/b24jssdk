## Common boot snippet

Every recipe starts with the same loader:

```ts
import { B24Hook, LoggerBrowser, type TypeB24 } from '@bitrix24/b24jssdk'

export function bootB24(): TypeB24 {
  const url = process.env.B24_HOOK
  if (!url) throw new Error('B24_HOOK env var is required')
  const $b24 = B24Hook.fromWebhookUrl(url)
  $b24.offClientSideWarning?.()
  return $b24
}

export const logger = LoggerBrowser.build('Recipe', process.env.NODE_ENV !== 'production')
```

Functions inside each recipe accept `$b24: TypeB24`, so the same code runs unchanged with `B24Frame` (in-iframe app) or `B24OAuth` (OAuth-installed app). Only the boot step changes.

## Notes that apply to every recipe

- All examples use the **v3** CRM API (`crm.item.list` + `entityTypeId` from `EnumCrmEntityTypeId`) for camelCase fields. Tasks (`tasks.task.*`), Disk (`disk.*`), and IM (`im.*`) stay on classic uppercase fields by necessity.
- Stage IDs on multi-funnel portals carry a category prefix (`C2:WON`, `C4:LOSE`). Recipes 1, 3, and 6 strip it via a small `baseStage()` helper.
- Filters use the **Bitrix24 prefix** style â€” `'>=opportunity'`, `'!stageId'`, `'%title'`. MongoDB-style `$gt`/`$ne`/`$contains` is a VibeCode-only convention and is **not** valid against Bitrix24 REST.
- Bitrix24 outbound webhooks always need a `200` response â€” recipe 7 always returns 200, even when the handler throws, so Bitrix24 doesn't retry for 24 h.
